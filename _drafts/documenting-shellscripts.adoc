== Doxygen all the things

{doxygen}[Doxygen] is generally a good starting point for every documentation approach, even more
when the targeted language is among the supported languages of the project.

Doxygen offers easy to understand {commands}[commands] to properly describe variables, functions
bodies and also complete files with every copyleft and `$ID$` information of your heart's desire.

I am quite certain you know them by heart, but let us quickly refresh our memory, so we are on the
same page:

[source,c]
----
 /** main
  * @brief Main function <.>
  *
  * @details <.>
  * Handles the main stuff
  *
  * @startuml <.>
  * main.c -> lang.c : get_lang()
  * @enduml
  *
  * @param[in]  argc  Number of arguments <.>
  * @param[in]  argv  Array with passed commandline arguments <.>
  * @retval  0  Default return value <.>
  **/

 int main(int argc, char *argv[]) {

    printf("Hello, %s", get_lang("NL"));

    return 0;
 }
----
. `@brief` is basically the headline of the block - so mostly the name of the function goes here
. Obviously, `@details` is intended to be more verbose, so feel free to add some prosa here
. This is a nice combination of sorts and allows to embed {plantuml}[PlantUML]
diagramsfootnote:[YES, two pain points at the price of one!]
. Next up are parameters with `@param` and also aa nice indication of the direction of access.
There aren't enough languages with pointers, so I am afraid just want to use it for the looks
. And finally the return value - which can be either verbose value by value or you can use just
`@return` and describe what is to be expected there

[NOTE]
There are different ways to prefix the commands:
The default is probably with a leading kbd:[/] ( backslash), but I prefer the kbd:[@]-version from
{javadoc}[Javadoc]

So with fresh memory let us quickly draft our Doxygen config:

[source,ini]
----
# Input/Output
INPUT             = src
OUTPUT_DIRECTORY  = doxygen
FILE_PATTERNS     = *.c *.h *.sh # <.>

# Special handling for shellscripts <.>
EXTENSION_MAPPING = no_extension=c
EXTENSION_MAPPING = sh=c

# Generators
GENERATE_HTML     = YES
GENERATE_LATEX    = NO
GENERATE_XML      = YES

# Project based
PROJECT_NAME      = showcase-asciidoxy
JAVADOC_AUTOBRIEF = YES # <.>
JAVADOC_BANNER    = YES
#RECURSIVE         = YES
EXTRACT_ALL       = YES #<.>

# Trick to force Doxygen not to ignore plantuml <.>
PLANTUML_JAR_PATH = /usr/share/java/plantuml.jar
----
<.> I like to suffix my shellscripts with  `.sh`, just to distinguish them from ordinary binaries,
so we need to add this to our pattern list
<.> Another trick up in our sleeves is the `EXTENSION_MAPPING`, which allows us to tell Doxygen to
treat unknown extension/files with no extension at all like C-files
<.> Yes, yes!
Give us the fully Javadoc experience!
<.> We just add `EXTRACT_ALL` as well to avoid nasty surprises later on
<.> And we also directly pin the PlantUML path, you never know

== Preprocessing from hell

I suppose the people behind Doxygen got lots of requests for fancy and esoteric languages, so
many in fact they couldn't keep up or were just satisfied with C.
Whatsoever, they've added {inputfilter}[input filters], which allow to prepare files before Doxygen
tries to make sense of them.

Gladly I am not the first one with this special interest and there is in fact a quite old Github
project, which provides the filter script of our dreams:

<https://github.com/Anvil/bash-doxygen>

I am not sure, if the name of the user or the used license is of any significance, but the whole
this was written in {sed}[sed] - yes just
sedfootnote:[I am proud of myself not yet to have made this pun..], but the FAQ brings everything
into perspective:

[quote,Anvil,'<https://github.com/Anvil/bash-doxygen#faq>']
+++
Q. Dude. sed ? Seriously ? A. Are you.. Jealous ?
+++

Messing with this script was a thrill-ride and highly educative.
In fact, I didn't know sed can be used in such a way and it felt a bit how I use
{awk}[awk].footnote:[While writing this I think in fact I might want to blog about these though.]

For the time being we just accept the script as-is, pretend we totally understand how it
works and treat it like AI-generated code and run it - {yolo}[--yolo]:

[source,shell]
----
$ doxygen/filter/doxygen-bash.sed src/build.sh | grep .

//!

----
<.> `$ID$` - see?
<.> This is just the way of telling Doxygen this is an uppercase variable of the type string
<.> This might be a bit weird, but the processing further down the pipeline requires
C-definitions and they don't cause any harm here, do they?
<.> Oh and we better strip the function bodies - less confusion for Doxygen is more good!
